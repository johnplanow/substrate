# A5.4 design — full close of red-team F3 (`--no-worktree` spec-tamper)

*Design-doc-first (per the ledger filing). Implementation is a future arc.
Captures the analysis so the next session can build without re-deriving it.*

## The finding (F3, HIGH)

`AcceptanceSpecCheck` (A1.3) catches spec tampering by comparing the story's
worktree copy of each acceptance spec (journeys.yaml, deferrals.yaml, the
profile acceptance block) against a TRUSTED copy. The trusted copy is read
from `trustedProfileDir` (the orchestrator's main tree). Under `--no-worktree`
dispatch, `workingDir === trustedProfileDir === projectRoot`, so the two sides
are the same file — the comparison is vacuous and an agent editing a spec in
the main tree is undetected. Ship 9 shipped a loud `acceptance-tamper-guard-disabled`
warning as the interim; this doc is the structural close.

## Why the obvious fix (`git show HEAD`) is WRONG

"Read the trusted side from the committed tree instead of the working tree"
fails because of commit-first (H0.1): the dev agent's output — INCLUDING any
spec edit — is auto-committed to HEAD *before* verification runs. Under
`--no-worktree` there is no separate branch, so commit-first advances the main
tree's HEAD. By the time `AcceptanceSpecCheck` executes, `git show HEAD:<spec>`
already contains the tamper → still vacuous.

## The correct baseline: the RUN-START SHA

Capture the trusted-tree SHA ONCE at orchestrator run start, before any story
runs or commits, and read the trusted spec side via `git show <runStartSha>:<path>`.
That baseline is immutable for the run's duration, so a spec edit committed
mid-run (in `--no-worktree`) or on a story branch (worktree mode) diverges from
it → caught in both modes. This also hardens the worktree-mode guard (a
bonus half-close of F4 for the guard inputs specifically).

## Implementation sketch (future arc)

1. **Orchestrator — capture `_acceptanceBaselineSha`.** At run start, gated on
   a committed registry existing (`git cat-file -e HEAD:.substrate/acceptance/journeys.yaml`)
   so registry-less runs pay no extra git call and the mocked-fs
   verification/H0.1 tests (which count `git rev-parse HEAD` calls) are not
   perturbed. When present: `_acceptanceBaselineSha = git rev-parse HEAD`.
   (Distinct from F4's `_runStartSha`, which is blocking-mode-only and feeds
   the audit/stage trusted LOADS; the guard applies in every mode, so it needs
   its own registry-gated capture.)
2. **Thread it** as `acceptanceSpecRef` into both `assembleVerificationContext`
   call sites.
3. **verification-integration — read the trusted side at the ref.** Replace the
   guard's `readOrNull(rel)` fs read with a `git show <ref>:<rel>` read (cwd =
   trustedDir) when `acceptanceSpecRef` is set; fall back to fs when absent or
   the git read fails (preserves current behavior + keeps mocked-fs tests
   green, since they provide no ref and no real git → fs fallback).
   Keep the no-value-import-from-sdlc discipline — use a local execSync git-show
   helper (the pattern already in this module for profile reads).
4. **Optionally** drop the Ship-9 interim `acceptance-tamper-guard-disabled`
   warning once the snapshot compare is armed in `--no-worktree` (or keep it as
   defense-in-depth if the container backend is still the real boundary).

## Empirical validation the implementation MUST carry (evidence rule)

Unit tests alone are insufficient for a coverage/guard change. Required:
- A matrix cell `spec-tamper-no-worktree`: stub edits a worktree/main-tree spec
  under `--no-worktree` blocking → `acceptance-spec-tampered` FAIL, no merge.
  (Mirror the existing `spec-tamper` cell with `--no-worktree`.)
- One live `--no-worktree` run on the fixture confirming a committed mid-run
  spec edit is caught against the run-start baseline.

## Residual (accepted, documented)

Even with the snapshot compare, `--no-worktree` remains weaker than worktree
isolation: the agent runs in the main tree with the (still-unscoped) Bash tool
and could, in principle, rewrite the run-start baseline's git objects or the
`.git` dir directly. That is the same H7 "worktree is accident-mitigation, not
a security boundary" envelope — the real boundary is the container backend
(`docs/2026-07-06-container-execution-seam.md`). A5.4 closes the accidental /
casual tamper vector under `--no-worktree`, matching the guarantee worktree
mode already provides; it does not claim containment against a hostile agent
with shell access. Recommend `--no-worktree` + `acceptance.mode: blocking`
stay a warned-against combination regardless.
