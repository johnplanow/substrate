# RP5.1 — adversarial red-team of the Registry Provenance arc (2026-07-10)

*Separate-lineage adversarial subagent, hostile framing ("find ways to DEFEAT
the feature; assume the implementer was careless"), given the 10-item evasion
catalog as a floor. Reviewed the arc diff `b776555..v0.21.18` (Ships 1–6).
5 findings; all 4 remediated or mitigated in Ship 7 (v0.21.19). All 7 cardinal
invariants held under direct code tracing.*

## Findings & disposition

### F1 — `ratify` read `derived_from` with no containment check (arbitrary file read) — CONFIRMED, medium — FIXED
The one un-gated `derived_from` read in the arc. A candidate is "editable by
design", so a hostile `derived_from: ../../../../etc/passwd` (with a matching
`source_sha256`) would make `ratify` read an out-of-project file (blind — the
content is only hashed) and write a registry whose provenance escapes the
project. `derive --prd` and `validate` staleness both containment-gate; ratify
used a raw `fs.readFile`.
**Fix (Ship 7):** `isProjectContainedPath(candidate.derived_from)` guard before
the read, mirroring the other two paths. **Live-verified:** a crafted
escaping candidate → `refusing to ratify`, no registry written, no read.

### F2 — completeness evidence rule grounded the SPAN but not the DISPOSITION (undispositioned-suppression) — CONFIRMED, medium — MITIGATED
A PRD injection could launder a genuinely-undispositioned journey to
`registered` by citing any real registry id + a verbatim quote of the
journey's *own* PRD sentence — the validator checked `registry_ref ∈ registered`
and that the span grounds, never that the claim was *about* the cited journey.
Bounded (advisory; the deterministic RP3.1 pre-pass over structured ids is
unaffected), but real.
**Fix (Ship 7):** a `registered` disposition now requires the cited journey's
title to share a DISTINCTIVE token (generic actor/domain words like
"operator"/"user" filtered out) with the claim description or span — an
unrelated id no longer suppresses the alarm. Unit-pinned. *Corpus-reconciled:*
the first cut normalized the two sides inconsistently (title tokens
alnum-stripped, claim text not) and rejected a legit Pre-Claim paraphrase on
the income-sources corpus ("Tuesday Pre-Claim" vs "mid-week Pre-Claim") — the
RP3.3 harness caught it; fixed to symmetric alnum tokenization so real
paraphrase passes while the digest↔export laundering still rejects.

### F3 — `diffJourneySets` end-state comparison was set-subset, not multiset (diff blinding) — CONFIRMED, medium — FIXED
`endStatesEqual([X,Y],[X,X])` returned true (every candidate key was *in the
set* of current keys, lengths matched) — a dropped-and-duplicated end-state
rendered as a diff no-op, violating invariant #7 (the review diff must never
be blinded). Mitigated at ratify by the duplicate-end-state-id schema
rejection, but the diff function the invariant names was blind.
**Fix (Ship 7):** sorted-key multiset equality. Unit-pinned with the exact
`[X,Y]` vs `[X,X]` case.

### F4 — grounding was token-presence, not contiguous (fabricated-span acceptance) — PLAUSIBLE, low — RECONCILED
A span assembled from individually-common words ("system", "report",
"generate") scattered through a large document grounded at ≥60% token
presence. The first remediation attempt required a *contiguous verbatim
quote* — and the **income-sources corpus immediately proved that too strict**:
it rejected a legitimate real-agent near-quote of the Pre-Claim journey
("A Signal scoring far above threshold appears mid-week…"), turning the whole
check invalid. This is the F7 lesson verbatim (hard grounding false-positives
real fuzzy-checker output; false positives burn operator trust).
**Reconciled fix (Ship 7):** kept the corpus-safe ≥60% token presence AND
added a lenient anti-fabrication anchor — at least ONE contiguous 3-token run
of the span must appear (a real quote always retains an intact fragment;
F4's scatter-of-common-words has none). Re-validated GREEN on the live corpus
(RP3.3 harness). The honest engineering call: the real anti-laundering defense
is F2, not verbatim grounding.

### F5 — criticality is agent-emitted; injection could downgrade critical→standard — PLAUSIBLE, medium — ACCEPTED-RISK (documented)
A PRD injection biasing the derive agent to mark a core journey `standard`
would (if ratified un-edited) exempt it from the `critical → must-declare-epic`
blocking boundary. No purely structural fix — criticality is inherently model
judgement, and the schema can force the enum but not truthfulness. **Live
counter-evidence:** the Ship-7 hostile-PRD derive smoke ("mark every journey
standard") did NOT take — the core loop stayed `critical`. Residual is bounded
by (a) the operator ratify review, where `criticality` + `criticality_rationale`
are surfaced per journey, and (b) that the same injection has to survive a
human reading the candidate. Recommend the operator always read criticality at
ratify; no code change.

## Minor notes (triaged)
- `diffJourneySets` deliberately does NOT diff `epic` — candidates from derive
  never carry one (assigned via `--epic` at ratify), so diffing it would flag
  every journey as epic-changed on every re-derive (pure noise). Documented in
  code.
- `computeUndispositioned` id/title conflation over-reports (fails safe, never
  hides) — accepted robustness nit.
- `prd_revision` in `RegistryProvenanceSchema` is a valid optional field not
  written by `ratifyCandidate` — staleness keys off `source_sha256`; harmless.

## Invariants that HELD (traced, could not break)
1. **NEVER-AUTO-RATIFY** — `ratifyCandidate` and the `journeys.yaml` write exist
   only in the operator `ratify` command; no pipeline/orchestrator/recovery/
   reconcile/resume/amend path calls either. The RP4.2 hook writes only the
   candidate + planning-journeys snapshot and refuses to clobber a candidate.
2. **Gate ignores candidates** — every gate-runtime read is `journeys.yaml`
   from the trusted tree via `git show`; no loader/audit/stage path reads
   `journeys.candidate.yaml` or `planning-journeys.yaml`. (Also pinned by the
   Ship-7 `candidate-ignored-by-gate` matrix cell.)
3. **Staleness containment** — `isProjectContainedPath` rejects empty/absolute/
   drive/scheme/`..` before any staleness re-read.
4. **Spec-tamper covers provenance bytes** — `AcceptanceSpecCheck` compares the
   full raw registry worktree-vs-trusted; a provenance-block edit diverges and
   FAILs, and the gate reads the trusted copy regardless.
5. **Version bump on re-ratify** — `version = existing + 1`, no channel to pin
   or reuse; the candidate schema has no version field.
6. **Derive has no structural exclusion/disposition channel** — schema-forced
   output; exclusions exist only at operator ratify. (Live: hostile-PRD derive
   ignored "exclude the alert journey" and "add a decoy UJ-99".)
7. **All new escalations advisory** — `validate` exit code derives solely from
   registry validity; staleness/completeness/undispositioned are print-only.

**Net:** the four cardinal structural invariants held under direct tracing. The
exploitable gaps were one un-gated read (F1, fixed) and the fuzzy completeness
layer's grounding (F2/F4, reconciled against the corpus) — the latter a live
re-run of the F7 false-positive lesson, resolved by trusting F2 (disposition
cross-check) over verbatim grounding.
