# Field feedback — income-sources first production run (2026-07-04)

Source: substrate 0.20.138 operating on the income-sources repo (externally-authored BMAD corpus: epics.md + sprint-status.yaml produced outside substrate's own planning phases, dispatched `--from` auto-detect with `--stories`). Operator: Claude Code session driving per the CLAUDE.md contract. First run: story 1-1 solo (calibration); second run: 1-2..1-5 batch (in flight when this was written).

## Findings, ranked

1. **Success without commit/merge (HIGH).** Run `5d3cf7ac` ended `pipeline:complete succeeded:["1-1"]`, story outcome `recovered` (2 review cycles, both `NEEDS_MINOR_FIXES`, then `story:auto-approved`) — but the worktree at `.substrate-worktrees/1-1` was left with **all work uncommitted** (branch `substrate/story-1-1` still at the base commit), and no merge to main occurred. The CLAUDE.md contract promises "agent's auto-commit lands on the branch" and "merge to main happens after verification SHIP_IT." Hypothesis: the `recovered`/auto-approved path (as opposed to first-class `SHIP_IT`) skips the commit+merge finalization. Operator had to hand-commit, hand-merge, and hand-update sprint state. `reconcile-from-disk` also isn't quite the right shape for this case — it marks Dolt state complete, but here Dolt was already complete and *git* was behind.
2. **`substrate init` did not create `.claude/commands/`** despite README/init output claiming `/substrate-run`, `/substrate-supervisor`, etc. Possibly related to the project's `.claude/` being gitignored or already populated (bmad skills live there).
3. **Telemetry `efficiency_scores` INSERT warnings** on pipeline start (`[telemetry-pipeline] [warn] {"err":{"sql":"INSERT INTO efficiency_scores (...` ) — repeated twice per run, appears cosmetic but noisy in the event stream.
4. **Cost display inconsistency**: `substrate report --run latest` header says `Cost: $0.0000` while the per-story row correctly shows `$0.3786`. Header aggregation appears broken (subscription-routed claude runs may be excluded from the header sum).
5. **Init droppings**: `AGENTS.md`, `GEMINI.md`, and `packs/` appeared untracked in the project root after init. Intentional? If so, the CLAUDE.md gitignore guidance should mention them; if not, they probably belong under `.substrate/`.

- **Spec fidelity**: the dev agent's AD-3 transition oracle matched the architecture spine's state diagram *exactly*, including amendments made hours earlier — context-free spec-following at the quality the pipeline promises.
- **Review loop caught real issues** on story 1: two `NEEDS_MINOR_FIXES` cycles + Tier-A recovery, no human attention needed until the landing check.
- Event stream (`--events` NDJSON) was clean to operate against; `status`/`health` JSON verdicts (`NO_PIPELINE_RUNNING`) are exactly what an agent operator needs.

## Root cause resolution (both create-story failure modes) + finding 10

Both `create-story-no-file` and the 10-minute `create-story-failed` timeout traced to environment auth: the operator's shell exports a stale `ANTHROPIC_API_KEY`, which spawned `claude` CLI sessions inherit — the CLI then rejects it ("auth source takes precedence over claude.ai login · Invalid API key") instead of using subscription auth. Confirmed: `env -u ANTHROPIC_API_KEY claude -p` works; third run relaunched with the var scrubbed.

10. **Auth failures are misclassified (HIGH).** An agent dying on "Invalid API key" surfaced as `create-story succeeded but returned no story_file path` (qualityScore 40) or as a full 600s timeout — never as an auth error. Two asks: (a) when `subscription_routing` is active / `api_billing: false` for the claude provider, scrub `ANTHROPIC_API_KEY` from the dispatch env so CLI subscription auth wins (config declares `api_key_env` — substrate knows exactly which var to control); (b) classify agent stdout matching known auth-error signatures as `auth-failure`, halt the run immediately (every subsequent dispatch will fail identically), and say so in the escalation. Would have saved ~25 minutes and two runs of confusion. Curious wrinkle for repro: the first two dispatches of the night (1-1, 1-2) succeeded from the same shell environment — auth failure onset was mid-run-1, suggesting the CLI's auth resolution isn't deterministic under this env or a session-reuse path masked it initially.

## Refined repro for finding 7 (supersedes the contamination hypothesis)

Run 3 (auth fixed) isolates it: **the first create-story dispatch of a run succeeds; every subsequent create-story in the same run fails in ~15s** with ~350 output tokens and "succeeded but returned no story_file path" (qualityScore ~40). Evidence: solo run `5d3cf7ac` (1-1 first → ok); run `0b56df1d` (1-2 first → ok, 1-3/1-4/1-5 → no-file); run 3 (1-3 first → full create/dev/review/verify success, 1-4/1-5 → no-file at 02:41:16 and ~02:44, ~15s each). Auth ruled out (1-3 succeeded in-run). Smells like a per-run resource consumed by the first create-story dispatch (prompt template/session slot/temp file) not reset for later ones. Workaround in production use: one story per `substrate run` invocation.

11. **Verification never executes the project's test suite (HIGH for quality claims).** Story 1-4 passed all six verification checks ("ALL PASS") while `pytest` had a genuinely failing test in the worktree (AD-13 absence-pause semantics). The `build` check reported `build-skip: no build command detected` for a uv/pyproject project and nothing else runs tests. For Python projects: detect `pyproject.toml` → `uv run pytest` (or honor a configurable verify command, like bmad-loop's `[verify] commands`). Without it, SHIP_IT/auto-approve verdicts assert quality that was never checked.

12. **Build-tool + language auto-detection is wrong for polyglot/Python repos (HIGH).** Story 2-2 escalated `build-failure: npm run build → Missing script "build"` on a Python/uv project. Two compounding issues: (a) the `build` verification check ran `npm run build` because (b) the dev agent had emitted spurious TypeScript files (`src/config/*.ts`) alongside the correct Python — and their presence flipped substrate's project-type detection to Node. Every real check passed (AC evidence, 7 runtime probes, source-ac-fidelity, tests). So a scope-contamination symptom (stray .ts) caused a false-negative build check that masked a genuinely-successful story. Asks: detect project type from the story's actual changed files + existing manifest (pyproject.toml present, no package.json) rather than any-file-triggers; and consider a verification signal for language/scope contamination (files in a language the project doesn't use).

13. **False-complete: story reported success with no implementation (HIGH).** Story 2-3 (retrieval client) returned status=complete, 0 review cycles, "239 tests pass" — but the branch contained ONLY the create-story spec file; zero implementation, zero new tests. The 239 were the pre-existing suite (so "tests pass" was true but vacuous — no new code to test). No verification check caught "story added no source/tests." Ask: a verification gate that fails when a non-trivial story's dev pass produced no net new source or test files (dev_story_signals.files_modified was effectively empty of code). Operator implemented it by hand per GOAL.md fallback.

14. **Inconsistent finalization: 2-5 auto-merged to main + cleaned worktree; earlier stories didn't (MED).** Story 2-5 self-completed the FULL cycle — dev, verify, fast-forward merge to main, worktree+branch removal — with no operator step. Correct outcome (334 tests green, no reversion), but it contradicts stories 1-1..2-4 which left uncommitted worktrees requiring manual land. Non-determinism in whether finalization runs makes an operator loop guess each time (I initially misread the missing worktree as a destructive false-complete). Ask: make finalization deterministic per config, and emit an explicit event (e.g. story:merged {sha}) so operators aren't inferring state from worktree presence. Sprint-status.yaml (the app's own tracking file, not substrate's) was NOT updated by the merge — expected, but worth documenting that substrate touches git, not the project's sprint ledger.

15. **Worktree isolation LEAKED into the main working tree (HIGH — data-integrity risk).** Story 3-3 escalated `merge-conflict-detected`, but the deeper problem: its dev work landed partially in the MAIN working tree, not just its `.substrate-worktrees/3-3` — untracked new files (machine/red_team/, tests/red_team/, a differently-named 0018 migration) AND uncommitted modifications to tracked files (machine/cli.py, machine/ledger/dossier.py). These were an EARLIER/intermediate version than the branch's committed 3-3 work (runner.py differed), so an operator merging naively would corrupt main. Recovery required: git clean the leak, discard the tracked-file leak (git checkout --), then merge the branch's real commit (which happened to fast-forward). Also: substrate's own self-merge attempt hit the migration-number collision (0018 in both) and escalated rather than renumbering. Asks: (a) guarantee worktree writes never touch the parent working tree; (b) auto-renumber colliding sequential migrations, or detect+escalate with the specific collision; (c) the escalation said "merge-conflict" but the real state was "parent tree dirtied by this run" — surface that.

16. **Architectural contamination: parallel package + DB layer, self-merged (HIGH).** Story 4-2 built the action endpoint under `src/actions/` (not the project's single `machine/` package) with its OWN db.py (duplicate connect/open_db/run_migrations), its OWN migration system (`src/actions/migrations/001_*.sql`), and a separate `decisions` table — violating AD-1 (one package), AD-2 (ledger is the only channel), AD-12 (decisions is a canonical ledger table). It also created a `setup_flags` table when the design stores that on the machine-state singleton. All 703 tests passed and it SELF-MERGED to main — the endpoint would have written decisions to a different SQLite file, invisible to the pipeline. This is the 2nd `src/`-contamination instance (2-2 emitted stray TypeScript `src/`). Substrate has no signal for "does this story's structure match the project's established package layout / persistence architecture." Ask: a verification check that flags new top-level source roots or new DB/migration layers inconsistent with the existing project. Operator reverted (clean FF reset) and re-implemented via a constrained agent.

17. **Leak severity escalates when the story's work is UNCOMMITTED (finding #15 addendum, CRITICAL).** Story 4-3 escalated (verification-failed) AND leaked into the main working tree — but this time substrate had NOT committed the dev work to the story branch (branch ref == main's HEAD). So the ONLY copy of 4-3's 21-file implementation was the uncommitted working-tree leak. An operator applying the finding-#15 recovery (clean the leak + `worktree remove --force`) then permanently DESTROYS the work — which is what happened here; 4-3 had to be re-run from scratch. Two asks, both important: (a) NEVER leave dev work uncommitted — commit to the story branch before signalling completion/escalation, so the branch is always the recoverable source of truth; (b) NEVER write into the parent working tree from a worktree run (the root bug). Operator-side mitigation added: before any leak cleanup, verify `git log <branch>` shows a real feat commit distinct from main HEAD; if not, the working-tree copy is the only copy — stash/commit it before cleaning.

18. **JS/TS contamination COMMITTED to main + node_modules bloat (HIGH, addendum to #12/#16).** Discovered 1885 tracked files under node_modules/ and dist/ (compiled JS incl. dist/red-team/prompt.js) on main — a substrate story emitted a TypeScript build (tsc/vite → dist/, npm install → node_modules) into a Python/uv project and it got merged. Story 4-3 did it a THIRD time (dist/db, dist/decisions, dist/rubric + a tracked node_modules). Pattern: substrate treats ambiguous stories as polyglot, scaffolds a JS toolchain, and its own npm-build check then "passes/fails" on that phantom toolchain (#12). Operator purged + gitignored. Ask: hard project-type lock — once a project is established Python/uv (pyproject.toml, no package.json), NEVER emit JS/TS/node artifacts or run npm; treat their appearance as a scope violation that fails verification.

19. **OPERATOR PROCESS FIX (self-inflicted, twice): verify commit before worktree removal.** Twice (stories 4-3, 5-1) substrate left a story's work uncommitted on its branch (finding #17). Recovering, I ran `git commit` then `git worktree remove --force` — but when substrate hadn't staged the files, the commit was a no-op ("nothing added") and the force-remove destroyed the work. New operator rule: after committing recovered work, ALWAYS confirm `git log <branch> --oneline -1` shows the NEW commit (SHA changed) BEFORE any `worktree remove --force` or merge; if the commit is a no-op, `git add -A` (excluding node_modules/dist) and retry. Root substrate cause unchanged (never leave dev work uncommitted, per finding #17). Cost: 2 stories re-run.

Recovery: operator staged the leak onto a branch, tested (936 green), merged. Non-interactive mode must either commit-before-escalate or never touch the parent tree.
