# Story 37-2: `substrate init` â€” Profile Detection & User Confirmation

## Story

As a developer initializing Substrate on a new project,
I want `substrate init` to auto-detect my project stack and write a `.substrate/project-profile.yaml` file,
so that the pipeline receives correct build, test, and language instructions without manual configuration.

## Acceptance Criteria

### AC1: Auto-Detection Invoked During Init
**Given** `substrate init` is run in any project directory
**When** `runInitAction()` executes
**Then** `detectProjectProfile()` (from Story 37-1's `src/modules/project-profile/detect.ts`) is called with the project root and its result is used for the profile step

### AC2: Human-Readable Profile Display
**Given** a project profile has been detected (non-empty result)
**When** `substrate init` runs in human output mode (not `--output-format json`)
**Then** the detected configuration is printed to stdout including: project type (single/monorepo), build command, test command, and â€” for monorepos â€” the per-package path + language breakdown

### AC3: Interactive Confirmation Prompt
**Given** `substrate init` runs without the `-y` flag (interactive mode)
**When** a non-empty profile has been detected and displayed
**Then** the user is prompted with `Accept detected project profile? [Y/n]`; if the user declines, the profile file is not written and init prints a message instructing the user to create `.substrate/project-profile.yaml` manually

### AC4: Non-Interactive Bypass
**Given** `substrate init` runs with the `-y` flag
**When** a profile is detected
**Then** the profile is written to `.substrate/project-profile.yaml` without any prompt

### AC5: Profile Written at Canonical Path
**Given** profile detection succeeds and write is confirmed (or non-interactive)
**When** `runInitAction()` completes
**Then** `.substrate/project-profile.yaml` exists, is valid YAML, contains a `# Generated by substrate init` comment header, and its content matches the detected `ProjectProfile` structure

### AC6: Existing Profile Preserved Without `--force`
**Given** `.substrate/project-profile.yaml` already exists
**When** `substrate init` runs without `--force`
**Then** the existing file is left unchanged and stdout shows `  .substrate/project-profile.yaml already exists â€” skipping (use --force to overwrite)`

### AC7: Graceful No-Detection
**Given** `detectProjectProfile()` returns `null` or a profile with no recognizable stack markers
**When** `substrate init` runs
**Then** the profile step is skipped entirely and â€” in human mode â€” stdout prints `  No project stack detected. Create .substrate/project-profile.yaml manually to enable polyglot support.`

## Dev Notes

### Architecture Constraints
- **Modular Monolith (ADR-001)**: profile writer lives in `src/modules/project-profile/writer.ts`; init-specific display helpers stay in `src/cli/commands/init.ts`
- **ESM import style**: all local imports use `.js` extension (e.g., `../../modules/project-profile/detect.js`)
- **Dependency on Story 37-1**: `detectProjectProfile()` and the `ProjectProfile` type must exist in `src/modules/project-profile/detect.ts` before this story is implemented; do not inline detection logic here
- **`js-yaml`** is already a project dependency â€” use it for YAML serialization (same as `init.ts` uses it for config files)
- **No new npm dependencies** â€” writer uses `fs/promises` and `js-yaml` only
- **Profile path**: always `join(substrateDir, 'project-profile.yaml')` where `substrateDir = join(dbRoot, '.substrate')`; `dbRoot` is already resolved via `resolveMainRepoRoot` in `runInitAction()`

### Key Files
- `src/cli/commands/init.ts` â€” add detection, display, confirmation, write steps inside `runInitAction()`
- `src/modules/project-profile/writer.ts` â€” new file (profile YAML serializer)
- `src/modules/project-profile/detect.ts` â€” provided by Story 37-1 (import only, do not modify)
- `src/modules/project-profile/project-profile.ts` â€” `ProjectProfile` type, provided by Story 37-1

### Insertion Point in `runInitAction()`
Insert profile detection as a new step between Step 1 (config files written) and Step 2 (scaffold BMAD framework). Comment label it `// Step 1b: Detect and write project profile`. This ordering ensures `substrateDir` is already created before writing to it.

### Interactive Pattern Reference
Follow the existing `promptSubscriptionRouting()` pattern: use Node.js `readline.createInterface` with `process.stdin`/`process.stdout`, return a typed promise, close the interface before resolving. Guard with `if (nonInteractive) return true` at the top.

### Testing Requirements
- Coverage threshold: 80% (enforced by vitest)
- Run `npm run test:fast` during iteration â€” targets unit tests only, no e2e
- For init-specific tests, look at existing tests in `src/cli/commands/init.test.ts` for the mocking pattern used for `fs/promises` and `AdapterRegistry`
