# Story 30-5: Pipeline-Time Recommendation Generation

## Story

As a pipeline operator running multi-story pipelines,
I want telemetry recommendations from story N persisted before story N+1's dispatch begins,
so that story N's optimization insights are queryable immediately and available for prompt injection in story 30-6.

## Acceptance Criteria

### AC1: BatchBuffer exposes a force-flush method that preserves the interval timer
**Given** a `BatchBuffer` with buffered items and an active interval timer
**When** `flush()` is called
**Then** all buffered items are emitted via the `'flush'` event immediately, the buffer is cleared, and the interval timer continues running (not stopped) — subsequent items can still be buffered and flushed normally

### AC2: IngestionServer exposes flushAndAwait() that force-flushes and awaits in-flight batches
**Given** the IngestionServer has a wired TelemetryPipeline with buffered items and/or in-flight `processBatch()` calls
**When** `flushAndAwait()` is called
**Then** `_buffer.flush()` is triggered immediately AND the method awaits all tracked `_pendingBatches` promises, resolving only when all in-flight telemetry processing is complete

### AC3: IngestionServer.flushAndAwait() is a no-op when no pipeline is wired
**Given** the IngestionServer has no TelemetryPipeline (stub/no-pipeline mode)
**When** `flushAndAwait()` is called
**Then** the method resolves immediately without error and without touching `_pendingBatches`

### AC4: Orchestrator calls flushAndAwait() at the inter-story boundary
**Given** the orchestrator is processing a sequential group of stories with an active IngestionServer
**When** story N's `processStory()` completes
**Then** `await ingestionServer.flushAndAwait()` is called before story N+1's `processStory()` begins — this ensures all OTLP payloads from story N are fully processed and persisted before the next dispatch

### AC5: Inline post-SHIP_IT efficiency and categorization blocks are skipped when IngestionServer is active
**Given** the orchestrator has an active IngestionServer (indicating TelemetryPipeline is handling telemetry)
**When** a story reaches SHIP_IT or LGTM_WITH_NOTES verdict
**Then** the inline efficiency scoring block (reads `getTurnAnalysis` + calls `storeEfficiencyScore`) and the inline categorization block (reads `getTurnAnalysis` + calls `storeCategoryStats` / `storeConsumerStats`) are both skipped — guarded by `if (ingestionServer === undefined)` — preventing duplicate DB writes and avoiding races against the async pipeline

### AC6: After flushAndAwait(), recommendations are queryable for the completed story
**Given** story N's OTLP batch was received by the IngestionServer and contained log-derived turns that produce at least one recommendation
**When** `ingestionServer.flushAndAwait()` resolves after story N completes
**Then** `telemetryPersistence.getRecommendations(storyKeyN)` returns a non-empty array — the recommendations generated by TelemetryPipeline's `_processStoryFromTurns` are durably stored

### BatchBuffer.flush() implementation

The `_flush()` private method already handles emptying the buffer and emitting the `'flush'` event. The new public `flush()` is a one-liner:

```typescript
/**
 * Trigger an immediate flush of buffered items without stopping the interval timer.
 * Use this to force-flush between pipeline phases while keeping the timer active.
 */
flush(): void {
  this._flush()
}
```

This is intentionally minimal — `_flush()` already does the right thing. The distinction from `stop()` is that `stop()` clears the timer while `flush()` does not.

### IngestionServer.flushAndAwait() implementation

`_pendingBatches` is already a `Set<Promise<void>>` on the class (see line 96 of ingestion-server.ts). The `stop()` method already uses `await Promise.all([...this._pendingBatches])`. The new `flushAndAwait()` reuses the same pattern but does not stop the server or the timer:

```typescript
/**
 * Force-flush buffered OTLP payloads and await all in-flight processBatch() calls.
 * Call this between story dispatches to ensure story N's telemetry (including
 * recommendations) is fully persisted before story N+1 begins.
 *
 * No-op when no TelemetryPipeline is wired.
 */
async flushAndAwait(): Promise<void> {
  if (this._buffer === undefined) return
  this._buffer.flush()
  if (this._pendingBatches.size > 0) {
    await Promise.all([...this._pendingBatches])
  }
}
```

### Orchestrator inter-story flush placement

The flush must happen BEFORE the GC hint and sleep, so processing starts as early as possible:

```typescript
for (const storyKey of group) {
  await processStory(storyKey)
  // Flush OTLP telemetry between stories — ensures story N's recommendations
  // are persisted before story N+1's dispatch begins (prerequisite for 30-6).
  await ingestionServer?.flushAndAwait()
  // GC hint between stories — best-effort
  ;(globalThis as { gc?: () => void }).gc?.()
  const gcPauseMs = config.gcPauseMs ?? 2_000
  // ...
}
```

### Inline telemetry guard rationale

The blocks at lines ~1658 and ~1686 in `orchestrator-impl.ts` were added in stories 27-5/27-6 as the primary path for telemetry computation — they read `getTurnAnalysis(storyKey)` from DB and re-run the analyzers inline. With the TelemetryPipeline active (stories 27-12, 30-1 through 30-4), the pipeline ALSO runs efficiency scoring, categorization, consumer stats, and recommendations. Without the guard, both paths run and produce duplicate rows.

The guard `if (ingestionServer === undefined)` is correct because:
- `ingestionServer` presence ↔ TelemetryPipeline is wired (see orchestrator setup around line 2214)
- When no IngestionServer, inline blocks remain the only path (backward compatibility)
- When IngestionServer is active, `flushAndAwait()` ensures pipeline has run before next story — inline blocks are redundant

**Important**: verify that `getTurnAnalysis(storyKey)` in these blocks was reading zero turns in practice anyway (because the BatchBuffer hadn't flushed yet when the verdict was determined). The guard doesn't change observable behavior — it just makes explicit that TelemetryPipeline owns this work when active.
