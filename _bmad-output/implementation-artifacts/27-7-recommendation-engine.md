# Story 27-7: Recommendation Engine

Status: ready-for-dev

## Story

As a pipeline operator,
I want an automated recommendation engine that analyzes telemetry patterns and generates actionable insights,
so that I can quickly identify and fix token waste without manually interpreting raw efficiency metrics.

## Acceptance Criteria

### AC1: Recommendation Type Definition
**Given** the telemetry recommender module is imported
**When** a consumer references `Recommendation`
**Then** the type contains all required fields: `id` (16-char deterministic hex derived from ruleId + storyKey + actionTarget), `storyKey`, `sprintId` (optional), `ruleId` (one of 8 enum values: `biggest_consumers`, `large_file_reads`, `expensive_bash`, `repeated_tool_calls`, `context_growth_spike`, `growing_categories`, `cache_efficiency`, `per_model_comparison`), `severity` (`'critical'|'warning'|'info'`), `title`, `description`, `potentialSavingsTokens` (optional number), `potentialSavingsUsd` (optional number), `actionTarget` (optional string identifying the file/tool/model), and `generatedAt` (injected ISO timestamp)

### AC2: Consumer Rules (4 rules)
**Given** turn analysis, semantic categories, and consumer groups for a story available in `RecommenderContext`
**When** `Recommender.analyze(context)` runs the consumer rule set
**Then** it generates recommendations for: (a) `biggest_consumers` — names top 3 token-consuming operations with their percentage share of total tokens; (b) `large_file_reads` — flags any file-read operation (span where `operationName` matches `file_read`) with `inputTokens > 3000`, description includes suggestion to use line ranges; (c) `expensive_bash` — flags any bash/shell operation (span where `toolName` is `bash` or `execute_command`) with `outputTokens > 3000`, description suggests filtering output; (d) `repeated_tool_calls` — detects the same `actionTarget` read more than once across a story's child spans and suggests caching the result

### AC3: Context and Trend Rules (2 rules)
**Given** per-turn token analysis (with `isContextSpike` flags) and semantic category trends for a story
**When** `Recommender.analyze(context)` runs context and trend rules
**Then** it generates: (a) `context_growth_spike` — for each turn where `isContextSpike` is `true`, a recommendation naming the turn number, input tokens, and the top 3 child span names that contributed to the spike; (b) `growing_categories` — a recommendation for each semantic category whose `trend` is `'growing'`, naming the category and its recent-vs-older token ratio

### AC4: Efficiency Rules (2 rules)
**Given** efficiency score and per-model metrics available in `RecommenderContext`
**When** `Recommender.analyze(context)` runs efficiency rules
**Then** it generates: (a) `cache_efficiency` — if overall cache hit rate is <30%, lists the top 3 worst-performing operations (lowest individual cache hit rate) and includes `potentialSavingsTokens` computed as `totalCacheMissTokens × 0.5` (savings if hit rate reached 50%); (b) `per_model_comparison` — if more than one model appears in spans, flags the model with the lowest cache efficiency against the highest, with `actionTarget` set to the underperforming model name

### AC5: Severity Thresholds
**Given** a recommendation generated by any of the 8 rules
**When** severity is assigned
**Then** `'critical'` is used when the referenced operation accounts for >25% of the story's total tokens; `'warning'` when it accounts for >10%; `'info'` otherwise; `context_growth_spike` recommendations are always at minimum `'warning'` even if the spike turn is below the 10% threshold; `growing_categories` recommendations are always `'info'` unless the category exceeds 25% of tokens, in which case `'warning'`

### AC6: Dolt Persistence
**Given** a completed `Recommendation[]` for a story
**When** `telemetryPersistence.saveRecommendations(storyKey, recommendations)` is called
**Then** all recommendations are batch-inserted into the `recommendations` Dolt table in a single transaction (one commit per call, not per row), and `telemetryPersistence.getRecommendations(storyKey)` returns them ordered critical → warning → info (then by `potentialSavingsTokens` descending within each severity), with each row validated against `RecommendationSchema` on read

### AC7: Determinism and Snapshot Tests
**Given** a fixed set of telemetry inputs in `tests/fixtures/telemetry/sample-recommender-context.json`
**When** `Recommender.analyze(context)` is called with that fixture (with a pinned `generatedAt` timestamp)
**Then** the output matches the Vitest snapshot stored in `src/modules/telemetry/__tests__/__snapshots__/recommender.test.ts.snap`; recommendation IDs are stable across runs because they are derived via `crypto.createHash('sha256')` from `ruleId + storyKey + (actionTarget ?? '') + index`; no `Math.random()` or `Date.now()` is called internally

## Dev Notes

### Architecture Constraints
- **Constructor injection**: `Recommender` accepts `ILogger` only — no repository injected; it is a pure analysis service
- **No new npm dependencies**: ID generation uses Node built-in `crypto` (`import { createHash } from 'node:crypto'`)
- **Zod-first types**: define all schemas in `types.ts`; derive TypeScript types via `z.infer<>`; validate all Dolt reads with `RecommendationSchema.parse()`
- **Repository pattern**: `saveRecommendations` and `getRecommendations` added to existing `TelemetryPersistence` and `ITelemetryPersistence` from story 27-3; do NOT create a separate repository class
- **Parameterized queries only**: all SQL uses better-sqlite3 prepared statements; no string interpolation
- **Import order**: Node built-ins (`node:crypto`) first, third-party second, internal relative paths third — blank lines between groups

### Severity Assignment Algorithm
```typescript
private assignSeverity(tokenPercent: number): RecommendationSeverity {
  if (tokenPercent > 25) return 'critical';
  if (tokenPercent > 10) return 'warning';
  return 'info';
}
```
Override floors applied per-rule:
- `context_growth_spike`: `Math.max(severity, 'warning')` (treat severity as ordinal: info < warning < critical)
- `growing_categories`: floor is `'info'`; promote to `'warning'` only if pct >25

### Deterministic ID Generation
```typescript
private makeId(ruleId: RuleId, storyKey: string, actionTarget: string, index: number): string {
  return createHash('sha256')
    .update(`${ruleId}:${storyKey}:${actionTarget}:${index}`)
    .digest('hex')
    .slice(0, 16);
}
```
Call with `index` being the 0-based position within a rule's output (most rules produce 1 recommendation; repeated_tool_calls and biggest_consumers may produce multiple).

### RecommenderContext Assembly (for wiring)
The recommender does NOT fetch data itself. The caller (pipeline post-processing or CLI command handler) assembles `RecommenderContext` by calling:
- `telemetryPersistence.getTurnAnalysis(storyKey)` → `turns`
- `telemetryPersistence.getCategoryAnalysis(storyKey)` → `categories` (from 27-5)
- `telemetryPersistence.getConsumerGroups(storyKey)` → `consumers` (from 27-5)
- `telemetryPersistence.getEfficiencyScore(storyKey)` → `efficiencyScore` (from 27-6)
- `telemetryPersistence.getSpansForStory(storyKey)` → `allSpans`

Then passes all of these plus `storyKey`, `sprintId`, `generatedAt: new Date().toISOString()` into `Recommender.analyze()`.

### Edge Cases
- **Empty turns/spans**: all rules must handle empty arrays gracefully — return `[]` not throw
- **Single model**: `per_model_comparison` emits nothing; never compare a model to itself
- **Zero total tokens**: `biggest_consumers` and severity assignment guard against division-by-zero; `tokenPercent = 0`
- **Cache hit rate of NaN/undefined**: treat as 0 (worst case) for `cache_efficiency` rule

### Testing Requirements
- **Unit tests**: mock `ILogger` via `vi.fn()` stubs; no SQLite or Dolt touches; inject minimal `RecommenderContext` per test
- **Snapshot test**: use a realistic fixture with multiple turns, categories, consumers, and efficiency data; pin `generatedAt` to a fixed timestamp so IDs are stable
- **Integration tests**: optional for this story — coverage of `saveRecommendations`/`getRecommendations` via unit-level in-memory SQLite is sufficient; full Dolt integration tested in Epic 26 layer
- **Coverage**: ≥80% branch and line coverage for `recommender.ts` and the new methods in `persistence.ts`
- **Test naming**: `describe('Recommender')` → `describe('analyze()')` → `it('should ...')`
